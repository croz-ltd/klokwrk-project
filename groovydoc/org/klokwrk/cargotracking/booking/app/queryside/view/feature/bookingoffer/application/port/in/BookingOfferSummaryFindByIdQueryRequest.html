        • @jakarta.validation.constraints.Null
          String customerId

          Internal customer identifier.

          In inbound request it must be null.

          It is assigned during request processing based on corresponding userId. customerId is used at the database level, while the processing pipeline is responsible to map provided userId to customerId.

          We could introduce another class for complete separation of concerns, i.e., something like BookingOfferSummaryQuery (and corresponding BookingOfferSummaryQueryResult}). That BookingOfferSummaryQuery class would then contain customerId, but not userId property. Of course, we should also implement appropriate mapping for such a scenario.

          However, for simplicity and the smaller number of DTO classes, we have just added the customerId property to the already existing BookingOfferSummaryFindByIdQueryRequest class. In addition, by adding Null annotation, we ensure the end user does not specify it.
